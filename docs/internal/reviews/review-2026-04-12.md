# Code Review: smooth-mcp (2026-04-12)

Second-pass review after completing all 26 items from todo1. Focuses on documentation, code efficiency, maintainability, demos, and API polish.

## 1. Documentation

### 1a. README parameter tables are incomplete

The "Solver options" table (README lines 136-148) omits several `solve_mcp` parameters that were added during todo1:

- `linear_solver` (default `"dense"`) — the GMRES forward solver option
- `krylov_tol`, `krylov_maxiter`, `krylov_restart`
- `regularize` (default `1e-12`)
- `verbose`

A user reading only the README would not discover the GMRES forward solver or Tikhonov regularization.

### 1b. `smoothed_residual` requires pre-normalized `F_fn`

`smoothed_residual` is exported in `__init__.py` but does not call `_normalize_F`. A user passing a single-argument `F(x)` will get a `TypeError`. Either normalize internally or document the requirement.

### 1c. README doesn't clarify relationship between `solve_mcp` and `make_mcp_solver_diff` parameters

`solve_mcp` has `linear_solver`, `regularize`, `krylov_*` parameters. `make_mcp_solver_diff` has `adjoint_method`, `gmres_*`, `cg_*`, `precond`. The README doesn't explain which parameters belong to which solver or why.

### 1d. `differentiable_lcp.py` demo passes dead CG parameters

`demos/differentiable_lcp.py:28-36` — `cg_tol=1e-10` and `cg_maxiter=2000` have no effect since `adjoint_method` defaults to `"gmres"`. Misleading for users reading the demo.

## 2. Code Efficiency

### 2a. Mu-continuation loop does not early-exit on convergence

`core.py:323-333` — The outer loop breaks only when `mu <= mu_min`. If the problem converges at an intermediate mu step, the solver still runs all remaining steps. Each step incurs the overhead of calling `newton_solve` and evaluating the initial residual. An early convergence check could save significant work.

### 2b. `_EMPTY_THETA = jnp.zeros(0)` created at import time

`core.py:27` — This creates a JAX array before the user configures `jax_enable_x64`, so it will be float32. Can cause dtype mismatches if user arrays are float64.

## 3. Maintainability

### 3a. Positional argument fragility in `_make_newton_solver` call

`core.py:303-317` — `newton_tol` is passed positionally as the 5th argument to `_make_newton_solver`, where the parameter is named `tol`. If someone reorders parameters, this silently breaks. Should use `tol=newton_tol`.

### 3b. `dir_deriv = -jnp.sum(H**2)` is incorrect with regularization

`core.py:198` — With regularization, the Newton direction is `d = -(J + reg*I)^{-1} H`, not `d = -J^{-1} H`. So the actual directional derivative `nabla(phi) . d = H^T J d != -||H||^2`. The error is tiny at `reg=1e-12` but grows with larger `regularize` values. The Armijo condition uses this incorrect value.

### 3c. KKT demo uses non-standard MCP formulation

`demos/kkt_conditions.py:11-21` — Manually encodes `primal * dual` as part of F, making it a 4D problem with quadratic nonlinearity. A standard MCP formulation would use `F(x) = Qx + c` with `l=0, u=inf` and let the MCP structure enforce complementarity. The current formulation is valid but misleadingly complex and doesn't demonstrate the natural KKT-MCP connection.

## 4. Demos

### 4a. No demo uses `linear_solver="gmres"`

The matrix-free Newton-Krylov option is a significant feature with zero demo coverage.

### 4b. No higher-dimensional demo (>4D)

All demos are 1D, 2D, or 4D. There is no demo showing a problem with 10+ variables, which is the natural use case for `linear_solver="gmres"` and would demonstrate scalability.

### 4c. No demo uses `solve_mcp_diff`

Exported and documented but never demonstrated.

### 4d. Most demos don't showcase `MCPResult` fields

Only 2 of 7 demos print `result.converged`, `result.residual_norm`, and `result.num_steps`.

### 4e. No demo shows gradients through bounds

Bound gradients were added in todo1 item 25 but no demo exercises them.

## 5. Testing Gaps

### 5a. No tests for `solve_mcp_diff`

The convenience wrapper has zero test coverage.

### 5b. No tests for `adjoint_method="cg"`

All gradient tests use the default GMRES. The CG code path in the backward pass is untested.

### 5c. No tests for `precond` parameter

A user-supplied preconditioner is never tested.

### 5d. No test for GMRES forward solver + differentiable backward pass

`make_mcp_solver_diff` does not expose `linear_solver` at all (see 6a), so this combination is impossible to test or use.

### 5e. `_normalize_F` edge cases untested

No tests for functions with default arguments (e.g., `def F(x, theta=None)` — would be incorrectly wrapped to ignore theta), `*args`/`**kwargs`, or bound methods.

### 5f. Singular Jacobian NaN test is fragile

`test_convergence.py:45` asserts `jnp.any(jnp.isnan(result.x))`. This tests a failure mode as if it were a feature. A future improvement could make this test fail even though the code got better.

## 6. API Asymmetries

### 6a. `make_mcp_solver_diff` is missing forward solver parameters

`make_mcp_solver_diff` does not expose `linear_solver`, `regularize`, `krylov_tol`, `krylov_maxiter`, or `krylov_restart`. It always uses `linear_solver="dense"` and `regularize=1e-12`. A user needing GMRES in the forward pass of the differentiable solver has no way to get it.

### 6b. Inconsistent krylov parameter naming

`solve_mcp` uses `krylov_tol/maxiter/restart` (forward Newton). `make_mcp_solver_diff` uses `gmres_tol/restart/maxiter` (backward adjoint). Similar things named differently.

### 6c. `d_x0 = None` silently zeroes gradient

`core.py:469` — When `differentiate_through_x0=False` (default), `jax.grad` w.r.t. `x0` returns zero with no warning. The gradient is mathematically non-zero but reports as zero, a usability trap.

### 6d. `smoothed_residual` exported but requires two-argument F

As noted in 1b, this is inconsistent with `solve_mcp` which accepts both `F(x)` and `F(x, theta)`.
